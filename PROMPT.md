# Prompt 设计文档

Prompt 框架说明、版本迭代指南和优化建议。

## Prompt 框架概述

### 设计原则

1. **清晰的角色定义**: 明确 AI 的身份和职责
2. **结构化的输入**: 提供格式化的案件信息
3. **明确的输出格式**: 定义决策和理由的格式
4. **上下文丰富**: 包含相似案例和历史数据
5. **容错机制**: 处理异常情况

### 核心组件

```
┌─────────────────────────────────────┐
│      角色定义 (Role Definition)     │
├─────────────────────────────────────┤
│      背景信息 (Background)          │
├─────────────────────────────────────┤
│      任务说明 (Task Description)    │
├─────────────────────────────────────┤
│      输入格式 (Input Format)        │
├─────────────────────────────────────┤
│      输出格式 (Output Format)       │
├─────────────────────────────────────┤
│      示例 (Examples)                │
├─────────────────────────────────────┤
│      约束条件 (Constraints)         │
└─────────────────────────────────────┘
```

## 版本演进

### v1: 基础版本 (初始版本)

**特点**: 简单直接，覆盖基本需求

```
你是一个风险控制专家。

根据以下案件信息进行风险评估：
- 交易 ID: {transaction_id}
- 交易金额: {amount}
- 国家: {country}
- 描述: {description}

相似案例:
{similar_cases}

请提供:
1. 风险评分 (0-1)
2. 决策 (APPROVE/REJECT/REVIEW)
3. 理由

输出格式:
{
  "risk_score": 0.5,
  "decision": "APPROVE",
  "reasoning": "..."
}
```

**性能指标**:
- Override 率: 20%
- 平均审核时间: 3800ms
- 准确率: 80%

**问题**:
- 决策理由不够详细
- 未充分利用相似案例
- 缺少规则引擎信息

### v2: 优化版本 (改进)

**改进点**:
1. 增加了规则引擎信息
2. 改进了相似案例分析
3. 增加了交易历史分析
4. 更详细的决策理由

```
你是一个资深的风险控制专家，拥有 10 年的反洗钱经验。

你的职责是评估交易风险并做出决策。

案件信息:
- 交易 ID: {transaction_id}
- 交易金额: {amount}
- 国家: {country}
- 描述: {description}

规则匹配:
{rule_matches}

相似案例分析:
{similar_cases_analysis}

交易历史:
{transaction_history}

请进行以下分析:
1. 规则匹配分析: 哪些规则被触发，风险程度如何？
2. 相似案例对比: 相似案例的决策是什么，为什么？
3. 交易历史分析: 这笔交易与历史交易有什么异常？
4. 综合评估: 综合以上信息，风险评分是多少？
5. 最终决策: 是否批准？

输出格式:
{
  "rule_analysis": "...",
  "similar_cases_analysis": "...",
  "transaction_history_analysis": "...",
  "risk_score": 0.5,
  "decision": "APPROVE",
  "reasoning": "..."
}
```

**性能指标**:
- Override 率: 12%
- 平均审核时间: 3500ms
- 准确率: 88%

**改进**:
- Override 率下降 40%
- 准确率提升 10%
- 审核时间减少 8%

### v3: 增强版本 (当前)

**改进点**:
1. 添加了风险因子权重
2. 改进了决策链路
3. 增加了置信度评分
4. 更好的异常处理

```
你是一个资深的风险控制专家，拥有 15 年的反洗钱和欺诈检测经验。

你的职责是评估交易风险并做出高质量的决策。

【案件信息】
交易 ID: {transaction_id}
交易金额: {amount}
国家: {country}
描述: {description}

【规则引擎结果】
{rule_matches}

【相似案例分析】
{similar_cases_analysis}

【交易历史分析】
{transaction_history}

【决策框架】
请按照以下步骤进行分析:

1. 规则风险评估
   - 哪些规则被触发？
   - 每条规则的风险权重是多少？
   - 规则层面的风险评分是多少？

2. 相似案例风险评估
   - 最相似的案例是什么？
   - 相似案例的决策是什么？
   - 相似案例的风险评分是多少？
   - 相似案例对当前案件的启示是什么？

3. 交易历史风险评估
   - 这笔交易与历史交易有什么异常？
   - 异常程度如何？
   - 历史交易的风险评分是多少？

4. 综合风险评估
   - 综合规则、相似案例、交易历史的信息
   - 计算最终风险评分（0-1）
   - 评估决策的置信度（0-1）

5. 最终决策
   - 决策: APPROVE / REJECT / REVIEW
   - 理由: 详细说明决策的原因
   - 建议: 如果是 REVIEW，提供审核建议

【输出格式】
{
  "rule_risk_score": 0.6,
  "similar_cases_risk_score": 0.7,
  "transaction_history_risk_score": 0.5,
  "risk_score": 0.65,
  "confidence": 0.85,
  "decision": "REVIEW",
  "reasoning": "...",
  "suggestions": "..."
}

【约束条件】
- 风险评分必须在 0-1 之间
- 置信度必须在 0-1 之间
- 决策必须是 APPROVE、REJECT 或 REVIEW 之一
- 理由必须清晰、具体、可追溯
- 如果置信度低于 0.6，建议 REVIEW
```

**性能指标**:
- Override 率: 10%
- 平均审核时间: 3400ms
- 准确率: 90%
- 置信度: 0.87

**改进**:
- Override 率进一步下降 17%
- 准确率提升 2%
- 置信度提升 3%

## 版本迭代指南

### 何时创建新版本

1. **性能指标下降**: Override 率上升 > 5%
2. **准确率下降**: 准确率下降 > 3%
3. **新的业务需求**: 需要支持新的风险因子
4. **规则引擎更新**: 新增或修改了规则
5. **用户反馈**: 收到多个用户反馈

### 版本创建流程

1. **分析问题**:
   - 识别当前版本的不足
   - 收集用户反馈
   - 分析性能数据

2. **设计改进**:
   - 确定改进方向
   - 设计新的 Prompt 结构
   - 准备测试数据

3. **实现和测试**:
   - 创建新版本
   - 在测试环境验证
   - 对比性能指标

4. **灰度发布**:
   - 在 10% 的流量上测试
   - 监控性能指标
   - 逐步扩大范围

5. **全量发布**:
   - 激活新版本
   - 监控运行状态
   - 保留旧版本作为回退

## 优化建议

### 1. 风险因子权重优化

根据实际数据调整各个风险因子的权重：

```
规则风险权重: 40%
相似案例权重: 35%
交易历史权重: 25%
```

### 2. 相似案例利用

改进相似案例的选择和分析：

```
- 增加相似案例数量（从 5 增加到 10）
- 考虑多维度相似性（金额、国家、行业等）
- 分析相似案例的决策模式
```

### 3. 规则引擎集成

更好地利用规则引擎的结果：

```
- 规则优先级排序
- 规则冲突处理
- 规则覆盖率分析
```

### 4. 交易历史分析

深化交易历史的分析：

```
- 交易频率分析
- 交易金额趋势
- 交易地域分布
- 异常交易检测
```

### 5. 置信度校准

改进置信度的计算：

```
- 基于历史准确率校准
- 考虑数据质量
- 考虑规则覆盖率
```

## 常见问题

### Q1: 如何快速测试新 Prompt？

A: 使用 API 端点 `/api/prompts` 创建新版本，然后在 `/api/ai/analyze` 中指定 `promptVersion` 参数。

### Q2: 如何回退到旧版本？

A: 使用 API 端点 `/api/prompts/{version}/activate` 激活旧版本。

### Q3: 如何评估 Prompt 的性能？

A: 使用 `/api/analytics/prompt-comparison` 端点对比不同版本的性能指标。

### Q4: 如何处理 Prompt 的多语言支持？

A: 创建不同语言的版本（如 v3_zh, v3_en），在调用时指定语言版本。

### Q5: 如何优化 Prompt 的成本？

A: 
- 减少不必要的上下文信息
- 使用更简洁的语言
- 考虑使用更小的模型
